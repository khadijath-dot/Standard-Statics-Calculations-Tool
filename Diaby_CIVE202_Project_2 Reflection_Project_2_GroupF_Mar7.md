Project \#2 focused heavily on the use of Python packages, variables and units, and defining functions. In this project, we learned how to implement the numpy package further in our code. In the past, we imported numpy at the top of our code and didn’t have to worry about it later. In Project \#2, we had to call the numpy package (np in the code space) to use trig functions. Python does not know what cos, sin, or tan means, but when numpy is called, np.cos, for example, Python now knows that the coder wants the cosine function from the numpy package.

Another thing we learned was organizing and manipulating variables and units. The statics problems contained a lot of variables – known and unknown – which required us to define them correctly with their respective units. We used the unit registry (ureg) from the pint package so that the numerical values that belong to the variables have units. In Python, we learned how to properly code a number with its units.

One of the biggest challenges in Project \#2 was just accurately translating the statics solution to Python. It’s one thing to write out the solution for the problems, but it’s another to type up the code equivalent. You have to stay in the confines of Python and make sure that Python will understand what you’re coding. What takes a few lines handwriting will take more . Another challenge was unit handling. The last statics problem had us convert the angles to radians rather than keeping them in degrees like the other two problems.

An implication from this project is reusing functions. Instead of rewriting functions, we can just change numbers and not have to resolve the function each time. It’s time efficient and requires less work and saves up space being used for codes that can be shorter.
